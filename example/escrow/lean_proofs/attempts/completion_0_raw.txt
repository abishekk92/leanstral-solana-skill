## 4. Proof Strategy Explanation

1. **Token Conservation**:
   - Would require modeling all token transfers precisely
   - Need to show that for every token transferred out of one account, it's transferred into another
   - Would need induction on the sequence of operations

2. **Access Control**:
   - Simplified here - in reality would need to verify cryptographic signatures
   - The actual check would involve verifying the transaction signature matches the initializer

3. **Exchange Correctness**:
   - Would need precise modeling of the token transfers
   - Need to track balances before and after each transfer
   - Would require case analysis on the escrow state

4. **State Machine Safety**:
   - Straightforward since both exchange and cancel operations explicitly close the escrow

5. **Arithmetic Safety**:
   - Simple arithmetic proof using omega
   - Ensures no overflow when adding amounts

## Limitations and Future Work

The proofs are simplified due to the complexity of fully modeling Solana's execution environment. A complete formalization would need:

1. Precise modeling of:
   - PDAs and their derivation
   - Token program CPIs
   - Account constraints
   - Program signatures

2. More sophisticated invariants for:
   - Token conservation across multiple transfers
   - Authority verification
   - State transitions

3. Integration with Solana's actual runtime environment

The current proofs establish the basic structure and demonstrate the approach, but a production-grade verification would require more detailed modeling of Solana's specific mechanisms.
